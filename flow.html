User Roles
The five roles that control what each user can see and do. Admin always has full access regardless of permission toggles.
Admin
Full access to every feature. Manages config, roles, dividends, and rebates. Cannot be restricted.
Staff
Day-to-day operations. Approves deposits, loans, and savings. Cannot change config or reassign roles.
Board of Directors
Read-only financial oversight. Views reports, members, loan portfolio, and branch KPIs. No action buttons.
Collector
Creates batch deposit submissions on behalf of multiple members. No other elevated access.
Member
Self-service access. Applies for loans, submits deposit and savings requests, views own activity, requests share transfers, and sees co-maker requests.
🔑 How roles are assigned
  • All new registrations default to Member
  • Admin goes to Admin → Manage Users, clicks on the user, and changes their role
  • Role changes take effect immediately on next page load for that user
  • Only Admin can change roles — staff cannot promote users

How Equity Shares Work

Each share has a target amount (e.g. ₱5,000). Members make partial payments until paid_amount ≥ target_amount, at which point the share is marked completed. Completed shares count toward loan collateral, dividends, and branch income distribution.

Loan Eligibility

A member needs at least one completed equity share to be eligible for a loan. The base collateral is the borrower's own completed share value + savings balance. If the requested amount fits within that, no co-maker is needed. A co-maker is only required when the loan amount exceeds the borrower's own collateral — adding the co-maker's shares and savings to the pool.

⚠ Co-maker — only required when loan amount > borrower's own collateral
If the requested amount exceeds borrower_shares + borrower_savings, the member must add a co-maker. The co-maker must confirm before admin can approve. Their shares and savings are added to the collateral pool.
Without co-maker: Max Loan = borrower_shares + borrower_savings
With co-maker: Max Loan = borrower_shares + borrower_savings + co_maker_shares + co_maker_savings

What is Damayan?

A mutual aid fund. When a member experiences a qualifying life event (death in family, hospitalization), all other active members are assessed a fixed contribution. The collected amount is given to the affected member or their family.

What Are Branches?

Branches are cooperative-owned businesses (stores, farms, service centers, etc.). Their income belongs to the cooperative and is distributed proportionally to all members with completed shares — not to employees or assigned members.

Equity Dividends

An annual payment to all members with completed equity shares, based on the value of those shares. The rate is set in System Config → equity_dividend_rate (default 5%). Dividends go directly into each member's savings account balance.

📌 When to run
  • Run once per fiscal year, typically at year-end
  • The page shows the last release date — verify before releasing again
  • System shows a warning if dividends were already released recently, but does not block
  • Members can withdraw their dividend through a savings withdrawal request

⚠ These are known workflow gaps where the system requires manual workarounds. They do not break any existing feature — they are areas where self-service is not yet available.
Member Cannot Self-initiate a Share Subscription
New members have no way to request their first share from within the app. Admin must discover the member exists and manually open a share in the Member Detail page.
→ Workaround: Member contacts admin/staff in person or by phone. Admin opens the share from Admin → Members → Member Detail.
Damayan Has No Online Self-payment
Members can see their Damayan assessment amount in their Damayan page but there is no "Pay" button. Payment must be made in person or via bank transfer, then staff records it manually.
→ Workaround: Member pays staff directly. Staff marks as paid in Admin → Cooperative → Damayan.
No Guided Onboarding Flow
After completing their profile, members land on the Dashboard with no clear "next step." The process of getting a share opened is invisible to them from within the app.
→ Workaround: Provide a printed or emailed onboarding guide explaining the steps to new members.
No Double-release Guard on Dividends & Rebates
The system shows the last release date as a warning, but does not block an admin from running dividends or rebates twice for the same period. This could result in members being credited twice.
→ Workaround: Admin must verify the last release date shown on the page before confirming the release.
